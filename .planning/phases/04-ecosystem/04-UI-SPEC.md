> Visual and interaction contract for Phase 4: Ecosystem.
> Scope: (1) deal-lang.org Astro + Starlight documentation site; (2) CLI terminal interaction contract for `deal init`, `deal install`, and `deal check` error output.
> Not in scope: React components, shadcn, desktop UI (Phase 6).

## Spacing Scale

Starlight uses its own CSS custom properties for spacing. The following contract governs custom content areas that fall outside Starlight's automatic layout (e.g., the landing page hero, any custom `.astro` components).

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, badge padding |
| sm | 8px | Compact element spacing (e.g., inline code padding) |
| md | 16px | Default content element spacing |
| lg | 24px | Card/block padding, hero content gap |
| xl | 32px | Landing page section gap |
| 2xl | 48px | Major section breaks on landing page |
| 3xl | 64px | Page-level top/bottom padding on landing hero |

Exceptions:

- Touch targets on mobile nav: minimum 44px height (WCAG 2.5.5 AAA target, Starlight handles automatically)
- Code block line height: 1.6 (Expressive Code default; do not override)

## Typography

Starlight uses a system font stack and manages heading/body scales automatically. The following contract governs custom content (landing page hero, any bespoke `.astro` pages) and the CLI terminal output contract.

### Docs Site Typography

**Weight palette: 400 (regular) and 700 (bold). No other weights are permitted.**

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (regular) | 1.5 | Starlight default; do not override |
| Label / caption | 14px | 400 (regular) | 1.4 | Code labels, sidebar items |
| Heading (h2 and h3) | 20px | 700 (bold) | 1.25 | Starlight default; do not override. Distinguish h3 from h2 by color tint (`--sl-color-gray-2` for h3 vs. `--sl-color-white` for h2 in dark mode), not by size. |
| Display (landing h1) | 36px | 700 (bold) | 1.15 | Custom landing page hero only |

**Focal point:** The landing page h1 "DEAL" (36px / 700 bold) together with the primary CTA button form the visual anchor of the entire site. Every other typographic element — subtitle, nav, sidebar, prose — is visually subordinate. Do not apply 36px or the display treatment anywhere outside the landing hero.

### CLI Terminal Typography Contract

Terminal output follows `rustc`/`cargo` diagnostic conventions (locked since Phase 2, REQ-phase-2-5-cli-shell). This is a behavioral contract, not a visual one.

| Output type | Format | Example |
|-------------|--------|---------|
| Error | `error[E25xx]: <message>` in red | `error[E2500]: dimension mismatch — expected Mass, got Voltage` |
| Warning | `warning[Wxxxx]: <message>` in yellow | `warning: unused import 'deal.std.units.lb'` |
| Note/hint | `  = note: <message>` in blue | `  = note: run 'deal install' to resolve missing dependencies` |
| Success | `<command> complete` in green | `deal init: project 'my-project' created` |
| Info | plain text, no prefix | `Downloading deal-std v0.4.0 from github.com/deal-lang/deal-stdlib...` |

## Color

### Docs Site (Starlight Theme)

Starlight ships two built-in themes: `starlight` (default light/dark) and `rapide`. Phase 4 uses the **default `starlight` theme** with a single accent color override. No full custom theme in Phase 4.

| Role | Value | Usage | Source |
|------|-------|-------|--------|
| Dominant (60%) | Starlight `--sl-color-white` / `--sl-color-gray-6` | Page background, content surface (auto light/dark) | Starlight default |
| Secondary (30%) | Starlight `--sl-color-gray-5` | Sidebar background, nav background, card backgrounds | Starlight default |
| Accent (10%) | `#0E7490` (Tailwind cyan-700) | Active nav item, inline links, CTA button, "Try it" badge | Custom override — see below |
| Destructive | not applicable | No destructive actions on docs site | — |

**Accent override:** Apply via Starlight's CSS custom property in `src/styles/custom.css`:

```css
:root {
  --sl-color-accent: #0E7490;        /* cyan-700 — engineering tool feel */
  --sl-color-accent-high: #164E63;   /* cyan-900 — hover/active states */
  --sl-color-accent-low: #ECFEFF;    /* cyan-50 — subtle accent tints */
}
```

Accent is reserved exclusively for:
1. Active sidebar navigation item (left border indicator)
2. Inline hyperlinks within prose content
3. Primary CTA button on landing page ("Get started")
4. Code language badge on Expressive Code blocks for `.deal`/`.dealx` fences

Do NOT apply accent color to: headings, body text, code block backgrounds, table rows, or decorative elements.

Dark mode: Starlight handles dark mode automatically via `prefers-color-scheme` and its built-in theme toggle. The accent override above applies to both modes via Starlight's CSS variable system.

### CLI Color Contract

| Output type | ANSI color | Usage |
|-------------|------------|-------|
| Error label | Red (ANSI 31) | `error[...]` prefix |
| Warning label | Yellow (ANSI 33) | `warning[...]` prefix |
| Note/hint label | Blue (ANSI 34) | `= note:` / `= help:` prefix |
| Success | Green (ANSI 32) | Completion messages |
| Emphasis (code/path) | Cyan (ANSI 36) | Quoted identifiers, file paths in diagnostics |
| No color | default | `--no-color` flag must suppress all ANSI codes |

### CLI Copywriting Contract

| Scenario | Message | Format |
|----------|---------|--------|
| `deal init` success | `deal init: project '{name}' created in ./{name}/` then `Next: cd {name} && deal install` | Two lines; second line is a plain hint |
| `deal install` — missing dep | `error[E2402]: dependency '{name}' not resolved — run 'deal install'` | Error format; emitted by `deal check` when `.deal/deps/{name}/` is absent |
| `deal install` downloading | `Downloading {name} v{tag} from {url}...` | Plain info; no spinner in Phase 4 |
| `deal install` success | `deal install: {N} dependencies resolved, deal.lock updated` | Success format |
| `deal install` lockfile changed | `deal install: deal.lock updated — commit deal.lock to reproduce this build` | Success format with note |
| `deal check` dimension error | `error[E2500]: dimension mismatch — expected {expected_dim}, found {actual_dim}` with source span | Standard rustc-style diagnostic |
| `deal check` mixed-unit error | `error[E2501]: mixed-unit comparison requires explicit conversion — use {to_fn}(...)` with source span | Standard rustc-style diagnostic; includes conversion hint |
| `deal check` unknown unit | `error[E2502]: unknown unit '{name}' — is 'deal-std' installed? Run 'deal install'` | Error with actionable hint |
| `deal build --target reqif` success | `deal build: wrote build/reqif/model.reqifz ({N} requirements, {M} relations)` | Success with summary stats |
| Destructive actions | none in Phase 4 | `deal init` warns if directory exists but does not delete |

**Overwrite guard for `deal init`:** If the target directory already exists and is non-empty, emit:
```
error: directory './{name}' already exists and is not empty
  = note: remove it manually or choose a different project name
```

## Code Block Contract

This section governs code examples throughout the docs. It is the primary UI surface unique to this phase.

### Language Identifiers

| Language ID | Grammar Source | Used For |
|-------------|---------------|----------|
| `deal` | `vscode-deal/syntaxes/deal.tmLanguage.json` | All `.deal` source examples |
| `dealx` | `vscode-deal/syntaxes/dealx.tmLanguage.json` | All `.dealx` composition examples |
| `toml` | Starlight/Shiki built-in | `deal.toml` and `deal.lock` examples |
| `bash` | Starlight/Shiki built-in | CLI command examples |
| `xml` | Starlight/Shiki built-in | ReqIF XML structure examples |

### Fence Markers

| Marker | Meaning | CI behavior |
|--------|---------|-------------|
| ` ```deal ` | Valid DEAL — must parse | `deal parse` called; exit non-0 fails CI |
| ` ```deal error-expected ` | Intentional parse error | CI skips `deal parse` for this block |
| ` ```dealx ` | Valid DEALX — must parse | `deal parse` called; exit non-0 fails CI |
| ` ```dealx error-expected ` | Intentional parse error | CI skips `deal parse` for this block |

### Shiki Scope Gate

Any `.deal` or `.dealx` fenced block that renders with zero highlighted tokens (all tokens fall through as unstyled plain text) causes the CI build to fail. This indicates the custom grammar was not loaded. Detection mechanism: post-build script checks HTML output for Expressive Code blocks where the `deal`/`dealx` language label is present but all token spans carry no scope class.

## Interaction States

### Navigation

| State | Behavior |
|-------|----------|
| Active page | Left border accent (`--sl-color-accent`), bold label |
| Hover | Background tint (`--sl-color-gray-5`), no transition animation |
| Collapsed sidebar group | Disclosure arrow rotates 90deg (Starlight default) |
| Mobile nav | Hamburger toggle; full-height overlay (Starlight default) |

### Code Blocks (Expressive Code)

| Feature | Behavior |
|---------|----------|
| Copy button | Appears on hover at top-right of block |
| Language badge | Shown for `deal`, `dealx`, `toml`, `bash`, `xml` |
| Line numbers | Off by default; enable only for blocks referenced by line number in prose |
| Diff highlighting | Not used in Phase 4 |
| Filename labels | Used when showing file context (e.g., `// deal.toml`) |

### Search (Pagefind)

| State | Behavior |
|-------|----------|
| Trigger | Keyboard shortcut `Ctrl+K` / `Cmd+K` + click on search bar (Starlight default) |
| Results | Page title + excerpt (Pagefind default) |
| No results | "No results for `{query}`" (Pagefind default) |

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Color contrast | Starlight default theme meets WCAG 2.1 AA for body text; accent `#0E7490` on white gives 5.74:1 (AA pass) |
| Keyboard navigation | Starlight sidebar and nav are keyboard-navigable by default |
| Skip link | Starlight ships "Skip to content" link (do not remove) |
| Focus visible | Starlight applies visible focus rings; do not suppress `outline` in custom CSS |
| Alt text | All custom images on landing page must have descriptive alt text |
| Semantic HTML | Use Starlight's MDX components (`<Aside>`, `<Card>`, `<Tabs>`) over raw `<div>` where available |
