    **Decision: `deal parse --json` emits the raw alphabetical-keyed AST JSON returned from `deal_ast_json()` DIRECTLY to stdout. NO envelope wrapping. The D-32 envelope is reserved for `deal check --json` and the diagnostic-emission paths of other subcommands.**

    Rationale (locked here for SUMMARY citation):
    - D-32 defines the envelope as diagnostic-bearing: `{v, command, deal_version, diagnostics, summary}`. It is designed to multiplex many small diagnostics with a summary — the consumer pattern is "iterate diagnostics + check summary counts".
    - The AST is a payload, not a diagnostic. Wrapping the AST in a new `ast` field inside the diagnostic envelope mixes two contract surfaces (diagnostic stream + payload), creating an undocumented public contract that consumers (Phase 3 LSP, CI tooling) cannot anticipate from the D-32 schema alone.
    - The minimum-surface choice keeps the D-32 contract crisp: when `--json` produces an envelope, the envelope is purely about diagnostics. When `--json` produces a payload (parse AST, build SysML output), the payload is emitted raw (already D-18 alphabetical-keyed).
    - This mirrors `deal build --target sysml-v2`'s behavior: the SysML JSON is written to a file (D-24), NOT wrapped in an envelope. `deal parse --json` writes the AST to stdout, also NOT wrapped.
    - Option B (extend the envelope with an `ast` field) was considered. Rejected because it requires a CONTEXT.md amendment + ADR — work this revision was instructed NOT to introduce. If a future phase needs an `ast`-bearing envelope (e.g., LSP wants AST + diagnostics multiplexed in one response), that phase opens a new ADR.
