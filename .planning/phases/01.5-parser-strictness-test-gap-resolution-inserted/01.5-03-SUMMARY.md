### Task 2 — Encode m_pins table + enforcement loop (commit `b3c70d8`)

**`tests/unit/recovery_corpus.zig`** — appended `Pin` struct + 20-entry `m_pins` constant table + per-pin enforcement loop after the existing soft-60 % gate. Pre-existing logic (directory walk, ≥50-file gate, panic check, no-diag print, soft 60 % gate) UNCHANGED. Pin enforcement is purely additive.

Pin shape:

```zig
const Pin = struct {
    name: []const u8,
    expected_code: ?[]const u8,   // null = accepted-no-diag
    adr_basename: ?[]const u8 = null,
};
```

Every entry references `Codes.e_*` named constants — zero inline `"E0123"` literals in the pin table (the only `"E....."` quoted-form in the file is a doc-comment example explaining the policy).

Enforcement loop reads each m0X file FRESHLY (independent of the directory walk above), parses, then asserts the appropriate two-outcome disposition:

  - `expected_code != null` → first `diags.items[0].code` MUST string-equal the pin code (zero-diag is a FAIL).
  - `expected_code == null` → zero diagnostics MUST be emitted AND `cwd.statFile(io, ".planning/decisions/<adr_basename>", .{})` MUST succeed.

Failure messages include file name, expected code, and observed code (or count) per the threat-register's T-15-03-03 disposition.

**`.planning/decisions/ADR-phase-1.5-m01-sd10-optional-semicolon.md`** — new ADR (5 sections + references): what the fixture demonstrates, why the grammar permits it (SD-10), why no diagnostic is emitted, what Phase 2's semantic analyser should catch (nothing — file is well-formed), disposition (pin = null + this ADR), future work (if a future W05xx stylistic-warning lane is added, update the pin to that code).

## Empirical Divergences From Filename-Suggested Intent

The following pins differ from the hypothesis table in PLAN.md `<interfaces>` because the parser's first response on these files is the EXPECTED-TOKEN failure rather than the recovery-tier or lexer-tier code suggested by the filename. The pin records the observed truth; the recovery / lexer-tier codes typically fire LATER as secondary diagnostics but only the FIRST emitted code is pinned.

| Pin | Filename-suggested code | Empirically observed code | Why |
|-----|-------------------------|---------------------------|-----|
| m02 unclosed_brace | E0401 e_unclosed_brace | E0100 e_expected_token | Parser hits EOF expecting `}` before recovery-tier sync fires |
| m05 invalid_utf8 | E0001 e_invalid_utf8 (lexer) | E0101 e_expected_definition (parser) | Lexer-tier code may appear as secondary; parser's response is the FIRST diagnostic |
| m06 unterminated_string | E0002 e_unterminated_string (lexer) | E0100 e_expected_token (parser) | Parser-side surfaces unterminated value as expected-token failure |
| m07 unterminated_template | E0002 / E0005 | E0100 e_expected_token | Same pattern as m06 |
| m09 invalid_modifier | E0120 e_invalid_modifier | E0101 e_expected_definition | Unknown identifier inside `<<>>` falls through to top-level |
| m14 brace_inside_attr | E0401 e_unclosed_brace | E0100 e_expected_token | Inline-block parser hits expected-token first |
| m15 double_at_in_annotation | E0111 e_expected_identifier | E0101 e_expected_definition | Second `@` ends annotation prematurely; top-level expected-def fires |
| m17 keyword_as_ident | E0111 / E0120 | E0100 e_expected_token | def expects IDENT; reserved-keyword `part` fails the expected-token check first |

Each divergence is documented as an inline comment next to its pin in `recovery_corpus.zig` so a future reader can trace the empirical reasoning without re-running discovery. This matches the Wave-1 "empirical-truth" pattern established by Plan 01.5-01's `recovery_definition.zig` Case-C lock-in.
