# Phase 1.5: Code Review Report

**Reviewed:** 2026-05-20T22:16:58Z
**Depth:** standard
**Files Reviewed:** 13 (6 source + 7 tests + build.zig)

## Summary

The Phase 1.5 changes are well-engineered, defensively coded, and the
focus-area checks (#1 lexer/parser correctness; #3 PUBLIC C ABI lock;
#5 drain-clear; #6 build-step dependency chain; #7 memory leaks on error
paths) all pass. The new strictness emissions (E0003 wiring, E0121,
E0122) trace through the grammar correctly, the forward-progress
invariant is preserved in `parseArgList`, the `drainLexerErrors` helper
correctly nulls the lexer field, the build graph contains no race or
cycle, and the diagnostic messages are comptime-fixed strings with no
source-byte injection vector.

**No Critical findings.** Four Warning findings were identified —
the most consequential is **WR-01**: the
`property_span_containment.zig` walker treats `.redefinition` as a leaf
even though `ast.Redefinition.value: ?*Node` is a child Node pointer.
Zig's `union(NodeKind)` exhaustive switch enforces that every payload
arm exists but does NOT enforce that the arm body recurses into every
`*Node` child. The walker therefore satisfies the compiler's
exhaustiveness check while silently bypassing the span-containment
contract for one payload — exactly the regression class the property
test was created to catch. Mitigated today by the fact that the
`.redefinition` payload is declared but never constructed by any parser
path (verified by grepping `Redefinition\b` across `src/`), so the
defect is currently latent rather than active.

The other Warnings are: a CI-gate silent-skip risk when test-filter
substrings fail to match (**WR-02**), an m05-pin divergence between
the C-ABI parse path (E0001) and the direct-parse path used by
`recovery_corpus.zig` (E0101) (**WR-03**), and a test-only handle
leak window in `determinism_parse_twice.zig` between
`deal_parse(handle_a)` and `deal_free(handle_a)` (**WR-04**). All
five Info items document minor coverage/documentation gaps that do not
affect contract correctness today.

## Warnings

### WR-01: property.span_containment walker omits `.redefinition.value` child recursion

**File:** `tests/unit/property_span_containment.zig:241-245`
**Issue:** The walker's switch arm groups `.redefinition` with leaf
NodeKinds:
```zig
.type_annotation, .multiplicity, .modifier_list,
.specialization, .redefinition, .structural_relationship,
.doc_comment => {
    // Leaves — no child Nodes to recurse into.
},
```
But `ast.Redefinition` (src/ast.zig:357-361) has `value: ?*Node` — a
child Node pointer. Zig 0.16's `union(NodeKind)` exhaustiveness check
forces every payload variant to appear in the switch, but it does NOT
enforce that the arm body recurses into every `*Node` field of that
payload's struct. The walker therefore passes the compile-time gate
while silently failing the span-containment contract for one payload.

The defect is currently dormant because `.redefinition` is declared in
the NodeKind enum but never constructed by any code path — grep
`Redefinition\b` across `src/` returns only the declarations
(ast.zig:357 struct and ast.zig:595 union arm). The fixture
`synthetic_fixtures[1] "redefinition"` (lines 99-106) parses
`redefines base.sensor sensor : Lidar;` but produces a
`.operator_statement` node, not a `.redefinition` node — so the
Plan 01 SUMMARY's "NodeKind coverage map" claim that this fixture
exercises `.redefinition` is also incorrect (see IN-01 below).

**Fix:** Move `.redefinition` out of the leaves group and add an
explicit recursion:
```zig
.redefinition => |p| {
    if (p.value) |v| try visit(v, node.span);
},
```
Even though `.redefinition` is currently unproduced, the explicit arm
prevents a future Phase 2+ change that starts constructing
`.redefinition` nodes from silently regressing the span-containment
invariant for that payload.

### WR-02: phase-1.5 CI-authoritative gate vulnerable to silent-skip if test-filter substring stops matching

**File:** `build.zig:176-191` (NOTE block and `phase-1.5-gate` step)
**Issue:** The locked CI command (per the NOTE block at lines 176-182
and CONTEXT.md §"Test infrastructure & build wiring") is
```
zig build phase-1-gate \
  && zig build test -Dtest-filter=determinism \
  && zig build test -Dtest-filter=property
```
Zig's `-Dtest-filter=<substr>` filters by name SUBSTRING; if no test
name contains the substring, `zig build test` still exits 0 with zero
tests executed. So a future rename of `determinism.parse_twice` to
`abi.parse_twice` (or `property.span_containment` to
`structural.span_containment`) would silently turn the CI gate green
without those tests ever running. The single-command convenience
`phase-1.5-gate` is NOT vulnerable because it depends on `test_step`
unfiltered, so test name drift is caught by the resulting "test not
registered" compile-time failure inside `_all.zig`.

The Phase 1.5 SUMMARY observes that the three-step form is preserved
"because it makes determinism/property gating surface-grep-able in CI
logs" — but the grep-ability comes at the price of silent-skip risk.

**Fix:** Either (a) add a build-step assertion that `-Dtest-filter`
matched at least one test (e.g., capture run-artifact output and grep
for "N tests passed" with N >= 1), or (b) document in build.zig that
any rename of `determinism.parse_twice` or `property.span_containment`
MUST also update the locked CI command, or (c) prefer the convenience
`phase-1.5-gate` as the CI gate and demote the three-step form to a
human-grep aid.

### WR-03: recovery_corpus.zig m05 pin asserts a parser-path code (E0101) that production C-ABI callers never observe (E0001 fires first)

**File:** `tests/unit/recovery_corpus.zig:84` (m05 pin)
**Issue:** The pin says:
```
.{ .name = "m05_invalid_utf8.deal",
   .expected_code = Codes.e_expected_definition },  // E0101
```
This is correct for the test's direct-parse path (`parser.parseFile`
on line 267) because `parser.parseFile` has no UTF-8 validation. But
`lib.deal_parse` (src/lib.zig:137-147) rejects invalid UTF-8 BEFORE
parsing and emits `Codes.e_invalid_utf8` (E0001) as the first
diagnostic. So:
- Test path (`recovery_corpus.zig`): m05 first-code = E0101.
- Production C-ABI path (Rust FFI, Tauri editor): m05 first-code = E0001.

The pin contracts a behavior that production callers will never
observe. The Plan 03 SUMMARY documents this divergence honestly
("E0001 may appear as a SECONDARY diagnostic but is not the first
item"), but the divergence remains a contract-surface mismatch: the
pin protects against regressions on a code path that real callers
don't use.

**Fix:** Either (a) extend `recovery.corpus` to ALSO parse each `m_pin`
file via `lib.deal_parse_internal` (test-only) and assert the C-ABI
path's first-code, or (b) annotate the m05 pin explicitly noting that
the production first-code is E0001 (via C-ABI UTF-8 validation) and
the test asserts only the direct-parse-path code, or (c) move m05 to a
dedicated "parser-path-only" pin section so future readers do not
mistake the pin for a contract on the production diagnostic stream.

### WR-04: determinism_parse_twice.zig leaks handle_a if any `try` between deal_parse and deal_free fails

**File:** `tests/unit/determinism_parse_twice.zig:71-103`
**Issue:** Lines 71-103 create `handle_a` via `lib.deal_parse` but
explicitly free it on line 103. There is no `defer lib.deal_free(handle_a)`
between the parse and the explicit free, so if any of the seven `try`
statements on lines 72, 78, 83, 91, 92, 99 fail, the arena backing
`handle_a` is leaked. The arena is allocated by `std.heap.page_allocator`
(src/lib.zig:92), not by `std.testing.allocator`, so `std.testing.allocator`'s
leak detector will not flag the leak — the test process exits and the OS
reclaims the pages. Cleanliness issue only, not a correctness defect.

`handle_b` (lines 106-108) correctly uses `defer lib.deal_free(handle_b);`
immediately after the parse.

**Fix:** Restructure so `handle_a` cleanup is unconditional. Two
options:
```zig
// Option A: deferred free with a guard flag.
const handle_a = lib.deal_parse(...);
var handle_a_freed = false;
defer if (!handle_a_freed) lib.deal_free(handle_a);
// ... after dupe and before parse B:
lib.deal_free(handle_a);
handle_a_freed = true;
```
```zig
// Option B: inline scope.
const json_a, const diag_a = blk: {
    const handle_a = lib.deal_parse(...);
    defer lib.deal_free(handle_a);
    // ... pull JSON, dupe, return as tuple
    break :blk .{ json_a_dup, diag_a_dup };
};
```
Either pattern ensures the arena is released on every exit path.

### IN-01: Synthetic `redefinition` fixture in property_span_containment.zig does not actually exercise `.redefinition` NodeKind

**File:** `tests/unit/property_span_containment.zig:99-106` (synthetic
fixture) and `tests/unit/_all.zig`-derived Plan 01 SUMMARY claim
**Issue:** The fixture source `redefines base.sensor sensor : Lidar;`
parses through `parseOperatorStatement` (parser_deal.zig:1581) which
produces a `.operator_statement` node, not a `.redefinition` node. The
`.redefinition` NodeKind has no construction path anywhere in `src/`
(grep `Redefinition\b src/*.zig` returns only the type declaration and
union arm). The Plan 01 SUMMARY's NodeKind coverage map claims this
fixture exercises `redefinition`, but no `.redefinition` node is ever
visited by the walker. Combined with WR-01, this means the
`.redefinition` arm of the property walker is doubly unexercised.

**Fix:** Either remove the misleading coverage claim from the
SUMMARY, or actually construct a `.redefinition` node somewhere
(would require parser changes — Phase 2 scope, not 1.5).

### IN-02: Plan 02 SUMMARY claims E0003 was "broadened to cover both `/* */` AND `/** */` unterminated cases" but only `/* */` is wired

**File:** `src/lexer.zig:413-466` (`skipTrivia` vs `scanDocComment`)
**Issue:** The Plan 02 SUMMARY decision "E0003 semantic broadening: ...
broadens it to cover both `/* */` AND `/** */` unterminated cases" is
aspirational. `scanDocComment` (lexer.zig:668-683) still emits
`.unknown` on EOF and does NOT set the `unterminated_block_comment`
field. Only `skipTrivia`'s block-comment branch (lines 437-458) wires
E0003. Production behavior for unterminated `/** */` is unchanged
from Phase 1.

**Fix:** Either (a) wire `scanDocComment` to also set the lexer field
(would add a second drain path), or (b) update the SUMMARY's decision
list to reflect that broadening is documentary-only.

### IN-03: parseArgList double-leading-comma falls through to parsePrimary's else arm and silently absorbs the second comma as an identifier named `","`

**File:** `src/expr.zig:359-373` (m19 leading-comma gate) +
`src/expr.zig:278-284` (parsePrimary else arm)
**Issue:** For `f(,,a)`: line 360 catches the first `,`, emits E0122,
advances. Line 369 checks for `r_paren` — the next token is `,` not
`)`, so falls through to line 375 `parseExpression(parser, 0)`.
parseExpression → parsePrefix → parsePrimary → `parser.advance()` on
line 198 consumes the second `,` — and the `else` arm (lines 278-284)
returns an `.identifier` node whose `name` is the bytes of the `,`
token (`","`). So `f(,,a)` produces args = [identifier(","), a] with
exactly 1 E0122 diagnostic instead of 2. This is loose recovery, not
incorrect parsing — the parser does not infinite-loop and forward
progress is preserved — but a phantom identifier-named-"," member is
introduced.

**Fix (optional):** In parseArgList line 369, accept r_paren OR comma
to immediately re-fire the gate cleanly:
```zig
const after_first = parser.peek();
if (after_first.tag == .r_paren) return try args.toOwnedSlice(allocator);
if (after_first.tag == .comma) continue; // re-enter the gate
```
Lowest priority — the parser is malformed-input territory, and Phase 2
semantic analysis would reject the phantom identifier downstream.

### IN-04: m08 / m18 / m19 strictness contracts are not exercised against `.dealx` files

**File:** `tests/unit/strictness_m08_m18_m19.zig:70-175`
**Issue:** The three strictness test cases only read `.deal` fixtures
(`tests/malformed/m08_unterminated_block_comment.deal`,
`m18_dangling_dot.deal`, `m19_empty_arg_list_comma.deal`). The drain
call IS wired in `parser_dealx.parseDealxFile` (line 224) so
`.dealx` files with unterminated block comments WOULD emit E0003, but
no test exercises this path. Similarly, no `.dealx` test exercises
the dot-branch (m18) or arg-list (m19) gates, which sit in `expr.zig`
and are reached from both parsers.

**Fix (optional):** Add three `.dealx` parallel fixtures and three
parallel test cases. Low priority — the `parser_dealx` drain wiring
is symmetric with `parser_deal` and the expression parser is shared,
so the contract is implicit. But a Plan 2+ change that diverges the
two parsers could silently break `.dealx` strictness without any
test gate firing.

### IN-05: property.span_containment warns but does not fail on showcase diagnostics; weakened to log-only by design

**File:** `tests/unit/property_span_containment.zig:399-408` and
`429-433`
**Issue:** Showcase files are well-formed by Phase 1 contract; any
diagnostic emitted while parsing them is a regression. The property
test logs (`std.debug.print`) when diags.items.len > 0 but does NOT
fail. So a regression in well-formed parsing that surfaces as a
diagnostic on a showcase file would slip past this test — the
property test would still pass as long as spans are contained.

The choice is intentional per the file's doc-comment ("The property
test only asserts span containment — log to stderr but do not fail"),
and other tests (parser_deal.coverage, parser_deal.snapshot) do
enforce well-formedness on showcase files. So this is defense-in-depth
weakening, not a contract gap. Documented here as INFO so future
maintainers know to look elsewhere if a "showcase produces diags"
regression is being chased.

**Fix:** None required; existing coverage by parser_deal.coverage and
the snapshot tests is sufficient. If desired, escalate the warning
to a failure to provide a third independent gate.
