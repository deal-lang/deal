# Phase 01.5 Plan 02: Parser Strictness (m08 / m18 / m19) Summary

Closed the three Phase 1 zero-diagnostic strictness gaps surfaced by Validation Audit 2026-05-20 by emitting `E0003` / `E0121` / `E0122` from the lexer-drain path and the expression parser's dot branch + argument-list parser, plus pinned the contracts in a dedicated `strictness.m08*/m18*/m19*` BDD test file — all three malformed fixtures now end Phase 1.5 producing a defined diagnostic instead of silently parsing.

## What Was Built

### Task 1 — Allocate E0121 + E0122 (commit `7cbf263`)

Added two NEW public constants to `src/diagnostics.zig` `Codes` namespace within the D-16 parser-deal range (E0100..E0299), inserted directly after `e_invalid_modifier = "E0120"`:

```zig
// E0121 — `.` not followed by an identifier (member-access without member name)
pub const e_dangling_dot = "E0121";
// E0122 — function-call arg list contains a comma with no preceding/following expression
pub const e_empty_arg_comma = "E0122";
```

m08 (unterminated block comment) was DELIBERATELY NOT assigned a new lexer code — the existing `e_unterminated_comment = "E0003"` (declared at `src/diagnostics.zig:64` since Phase 1, but never emitted) is the canonical code per CONTEXT.md §specifics and is wired in Task 2.

### Task 2 — Implement m08/m18/m19 strictness (commit `b95e616`)

**Per locked CONTEXT decision: all three got IMPLEMENTATIONS, not ADRs.** The relevant grammar productions unambiguously reject the constructs the malformed fixtures exercise (`/* */` is a closed pair; `.` requires IDENT after; `_ArgumentList ::= Expression ( "," Expression )*` requires expressions between commas).

**m08 unterminated block comment — wires EXISTING E0003**:

- `src/lexer.zig` (lines 256-282 + skipTrivia ~414-434):
  - Added `unterminated_block_comment: ?ast.Span = null` field to the `Lexer` struct (16 bytes, arena-neutral).
  - In `skipTrivia`'s block-comment EOF branch, captures `comment_start: u32 = self.pos` BEFORE `self.pos += 2` consumes the `/*`. On the `else`-branch (EOF reached before `*/`), now records `unterminated_block_comment = Span{ comment_start, source.len }` instead of just advancing to EOF silently.
- `src/parser_deal.zig` (~line 300, before `syncToDefinition`):
  - New public `drainLexerErrors(p: *Parser) !void` helper. Appends an `E0003 e_unterminated_comment` diagnostic at the captured span and clears the lexer field (T-15-02-04 bounded-state mitigation prevents repeated drains from double-firing).
- `src/parser_deal.zig` (parseDealFile, ~line 405): Calls `try drainLexerErrors(p)` after the main parse loop terminates on EOF, before constructing the deal_file node.
- `src/parser_dealx.zig` (parseDealxFile, ~line 219): Calls `try parser_deal.drainLexerErrors(p)` at the equivalent end-of-loop position so `.dealx` files with unterminated block comments in either preamble or body also emit E0003.

**m18 dangling dot operator — emits NEW E0121**:

- `src/expr.zig` parseExpression dot branch (~lines 102-130):
  - Pre-Phase-1.5: `parser.advance()` returned the member token unconditionally and `parser.source[member_tok.span.start..end]` was sliced as the member name — silently accepting `a.;` as `member_access{ receiver: a, member: ";" }`.
  - Phase 1.5-02: now `parser.peek()` first; if `member_tok.tag != .ident`, emits `Codes.e_dangling_dot (E0121)` at `dot_tok.span` and `break`s the Pratt loop without advancing. The statement-tier sync at the enclosing context picks up the stray non-IDENT (e.g. `;`).

**m19 empty-arg comma — emits NEW E0122**:

- `src/expr.zig` parseArgList (~lines 316-380):
  - Pre-Phase-1.5: `parser.peek()` would see `.comma` and call `parseExpression(parser, 0)` anyway, silently producing an "identifier" arg whose name was `,`.
  - Phase 1.5-02: now handles all three shapes the grammar disallows:
    1. **Leading comma** `f(,a)`: emits at the leading comma's span, consumes it, falls through to parse the next expression.
    2. **Leading-only** `f(,)`: emits at the comma, consumes it, returns empty args when next token is `r_paren`.
    3. **Double comma** `f(a,,b)`: emits at the offending comma's span, `continue`s the loop (the next comma fires the gate cleanly).
    4. **Trailing comma** `f(a,)`: emits at the trailing comma's span and `break`s the loop (the `)` closes the call).

Every error branch consumes the offending comma BEFORE the next peek, preserving the forward-progress invariant (T-15-02-03 mitigation; mirrors Phase 1 LEARNINGS Pattern "Forward-progress single-advance fallback").

Imports added: `const diagnostics_mod = @import("diagnostics");` in expr.zig (didn't exist pre-Phase-1.5).

## Deviations from Plan

None substantive. Implementation tracked the plan's task structure exactly.

**Minor implementation choices left to executor discretion (per plan's "pick the option with the smallest diff" guidance):**

1. **Lexer-error surface mechanism:** Chose plan's Option A (a `Lexer.unterminated_block_comment: ?ast.Span` field drained by the parser) over Options B (returning `?Token`) and C (per-call helper). Option A is the smallest diff — one field + one drain site per parser file. Option B would have forced changes at all 13 `next()` caller sites; Option C would have required threading the collector through skipTrivia.

2. **Drain frequency:** Chose plan's "simplest correct wiring" — a single drain at parseFile-end rather than per-token. M08's shape places the unterminated `/*` as the LAST trivia before EOF, so a single end-of-loop drain suffices. The drain helper is `pub`, so a future plan can call it per-token if mid-file shapes appear.

3. **Trailing-comma policy:** The grammar (`spec/grammar/deal.ebnf` line 1680: `_ArgumentList ::= Expression ( "," Expression )*`) does NOT permit trailing commas. Locked trailing-comma as an error alongside leading and double commas. Documented in the decision list above so a future grammar revision can revisit cleanly.

4. **m08 message string:** Chose `"unterminated `/* */` block comment"` (per the plan's example) rather than the field's pre-Phase-1.5 documented doc-comment-only scope. The identifier `e_unterminated_comment` is comment-type-neutral; the message disambiguates.
