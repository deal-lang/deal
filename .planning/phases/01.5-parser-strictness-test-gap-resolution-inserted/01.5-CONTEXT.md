<domain>
## Phase Boundary

Phase 1.5 is a *remediation phase*, not a feature phase. Phase 1 shipped with all
automated gates green, but `/gsd:validate-phase 1` surfaced concrete test-quality
weaknesses that mean the parser contract is *intended* in places where it should
be *asserted*. Phase 2 (semantic analysis) will build directly on the AST shape
and diagnostic stream this parser produces — so the test contract must be
strengthened *before* Phase 2 adds new dependencies.

**In scope:**
- Replacing vacuous test assertions (`diags.items.len >= 0` and equivalents) with
  named-property checks.
- Pinning per-file expected primary diagnostic codes for the 20 hand-curated
  malformed files (m01..m20).
- Adding an automated determinism test (parse-twice → byte-identical JSON) that
  locks the D-18 alphabetical-key invariant.
- Deciding the fate of m08 (unterminated block comment), m18 (dangling dot
  operator), m19 (empty arg with comma): each either gets a defined diagnostic
  code, or is documented as accepted via an ADR.
- Adding a property-style test that asserts every AST node's span is contained
  within its parent's span across the 19-file showcase corpus.

**Out of scope:**
- New language features. The grammar (`lexical.ebnf`, `deal.ebnf`, `dealx.ebnf`)
  is frozen for this phase — m08/m18/m19 are decided by either picking from
  D-16's already-reserved error code ranges OR accepting via ADR.
- Performance work. Test additions may slow `zig build test` by tens of ms;
  acceptable.
- Refactoring the parsers, lexer, or json emitter. Strictness fixes are the
  smallest patches that produce the diagnostic.
- Touching the generator-produced malformed files (`gen_*.deal[x]`). Those
  remain under the soft 60 % gate; only the hand-curated `m01..m20` get
  per-file pins.
- Snapshot regeneration. The 19 AST + 19 token snapshots stay byte-identical;
  any phase-1.5 change that mutates a snapshot is a regression to investigate,
  not a snapshot to re-bless.

</domain>

<decisions>
## Implementation Decisions

### Test assertion strictness (REQ-phase-1.5-1)
- **Locked:** Every vacuous `>= 0` assertion against a `*.len` value must be
  replaced with a named-property check. Grep gate: `grep -nE 'len >= 0' tests/unit/*.zig`
  must return zero matches at phase exit.
- **Locked:** Each `recovery_*` test case asserts (a) the exact diagnostic code
  emitted (e.g. `expectEqualStrings("E0101", diags.items[0].code)`) and (b) the
  recovered AST shape at the expected position (e.g.
  `df.definitions[1].payload.part_def.name == "Good"`).
- **Locked:** Existing `parser_dealx.tag_balance` is the exemplar to mirror in
  `recovery_statement`, `recovery_definition`, `recovery_dealx_tag`.

### Malformed-file pinning (REQ-phase-1.5-2)
- **Locked:** Per-file expected primary diagnostic code declared for each of
  `m01..m20`. Use a Zig declaration table (`tests/malformed/_pins.zig` or
  equivalent constant block in `recovery_corpus.zig`).
- **Locked:** `recovery.corpus` fails if any hand-curated file emits a different
  primary code than its pin.
- **Locked:** 30 generator-produced files keep the soft 60 % gate — they are
  generative and per-file pinning would couple the test to mutation semantics.

### Determinism test (REQ-phase-1.5-3)
- **Locked:** New test `determinism.parse_twice` in
  `tests/unit/determinism_parse_twice.zig` parses each of the 19 showcase
  files twice in the same process; asserts byte-identical AST JSON.
- **Locked:** Test uses `deal_parse` (the public C ABI) not internal helpers —
  determinism must hold at the contract surface.
- **Claude's Discretion:** Whether to also assert byte-identity of the
  diagnostics JSON (probably yes; trivial cost).

### m08 / m18 / m19 disposition (REQ-phase-1.5-4)
- **Locked:** Each of the three files MUST end Phase 1.5 in exactly one of two
  states: (a) emits a defined diagnostic with a code from D-16's existing range,
  or (b) has an ADR at `.planning/decisions/ADR-phase-1.5-<topic>.md`
  documenting why the input is grammatically acceptable.
- **Diagnostic-code allocation policy (Claude's Discretion within these bounds):**
  - D-16 lexer range is `E0001..E0099` (currently allocated: E0001..E0005;
    next-available E0006). The unterminated-block-comment code `E0003`
    (`e_unterminated_comment`) ALREADY EXISTS in `src/diagnostics.zig` —
    Phase 1 declared it but the lexer never emits it. **m08 should re-use
    the existing E0003 by wiring its emission in `lexer.zig skipTrivia`'s
    block-comment path; do NOT allocate a new code.**
  - D-16 parser-deal range is `E0100..E0299` (currently allocated:
    E0100..E0103, E0110..E0111, E0120; next-available within the parser
    block is E0104 or E0121 — pick the slot that reads sensibly given the
    nearby codes). Allocate fresh codes for m18 (dangling dot operator)
    and m19 (empty argument with comma).
- **Authoritative range source:** `src/diagnostics.zig` lines 8-9 and 61-75
  declare the actual ranges and allocations. If this CONTEXT.md ever
  disagrees with `src/diagnostics.zig`, the source file wins.
- **Claude's Discretion:** Whether m18 dangling-dot is best modeled as
  "expected expression after `.`" (parser) or "stray `.` token" (lexer recovery)
  — pick whichever produces the cleaner span and message.

### Span-containment property test (REQ-phase-1.5-5)
- **Locked:** New test `property.span_containment` in
  `tests/unit/property_span_containment.zig` walks every AST node of every
  showcase file (19 total) and asserts
  `child.span.start >= parent.span.start && child.span.end <= parent.span.end`.
- **Locked:** Must pass against the currently-locked snapshots without any
  snapshot regeneration. If it fails, that is a Phase 1 bug being surfaced
  retroactively and gets fixed in Phase 1.5.
- **Claude's Discretion:** Whether to also include the synthetic fixtures from
  `parser_deal.coverage` in the property walk (probably yes; covers NodeKinds
  not present in the 19-file corpus).

### Test infrastructure & build wiring
- **Locked:** All new tests register through `tests/unit/_all.zig` (the
  established umbrella convention).
- **Locked:** All new tests must be runnable via
  `zig build test -Dtest-filter=<name>` (per Phase 1 sampling-rate contract).
- **Locked:** Phase 1.5 exit gate command:
  `zig build phase-1-gate && zig build test -Dtest-filter=determinism && zig build test -Dtest-filter=property`
  all exit 0.

### Recovery test strengthening — non-goals
- **Claude's Discretion:** Whether to add *new* recovery cases or just
  strengthen the existing 9 cases (3 in statement + 3 in definition + 3 in
  dealx_tag). Strengthening is mandatory; new cases are bonus. Prefer
  strengthening unless a strictness fix in REQ-phase-1.5-4 exposes a gap.

## Specific Ideas

### Suggested code allocations (verified against current `src/diagnostics.zig`)
- **m08 unterminated block comment** → **re-use existing `E0003`
  `e_unterminated_comment`**. The code is already declared in
  `src/diagnostics.zig:64` but the lexer never emits it; Phase 1.5 wires
  the emission in `lexer.zig skipTrivia`'s block-comment path. No new code.
- **m18 dangling dot operator** → allocate `E0121`
  (parser-deal range E0100..E0299; next-available in the parser block is
  E0121 since E0120 is the last existing).
- **m19 empty arg with comma** → allocate `E0122`.

D-16 ranges (per `src/diagnostics.zig:8-9`): `E0001..E0099` lexer,
`E0100..E0299` parser-deal. Use the diagnostics.zig file as the
authoritative source — if any earlier reference disagrees, prefer the
source file.

### Exemplar BDD pattern to mirror
From `parser_dealx_tag_balance.zig:55-69` — exact code, exact count, exact
primary-span byte offset, exact secondary-span byte offset, secondary-label
substring check. Phase 1.5 strengthens 9 recovery cases to this standard.

### Determinism test concrete shape

**MUST use the public C ABI `deal_parse` / `deal_ast_json` / `deal_free`** so
the determinism contract is asserted at the surface external callers
(Rust CLI, Tauri desktop) will use. `deal_parse_internal` is forbidden in
this test — internal-helper byte-equality could pass while the public
ABI's JSON cache produces different bytes.

```zig
test "determinism.parse_twice" {
    const gpa = std.testing.allocator;
    for (showcase_files) |rel| {
        const path = try std.fmt.allocPrint(gpa, "tests/showcase/{s}", .{rel});
        defer gpa.free(path);
        const source = try cwd.readFileAlloc(io, path, gpa, .unlimited);
        defer gpa.free(source);

        // First parse — capture AST JSON via C ABI.
        const handle_a = lib.deal_parse(source.ptr, source.len,
                                        rel.ptr, rel.len);
        defer lib.deal_free(handle_a);
        var ptr_a: [*]const u8 = undefined;
        var len_a: usize = 0;
        _ = lib.deal_ast_json(handle_a, &ptr_a, &len_a);
        const json_a = try gpa.dupe(u8, ptr_a[0..len_a]);
        defer gpa.free(json_a);

        // Second parse of the same source — must produce byte-identical JSON.
        const handle_b = lib.deal_parse(source.ptr, source.len,
                                        rel.ptr, rel.len);
        defer lib.deal_free(handle_b);
        var ptr_b: [*]const u8 = undefined;
        var len_b: usize = 0;
        _ = lib.deal_ast_json(handle_b, &ptr_b, &len_b);

        try std.testing.expectEqualStrings(json_a, ptr_b[0..len_b]);
    }
}
```

The `gpa.dupe` of the first JSON is required because `deal_free(handle_a)`
will invalidate `ptr_a` before the second parse completes; copy into a
gpa-owned buffer so the comparison sees both byte strings live
simultaneously.

### Span-containment test concrete shape

## Deferred Ideas

### Generative property tests
Fuzz-style "any random well-formed identifier roundtrips through
lex→parse→emit→lex" — useful but out of scope for 1.5. Capture as a
post-Phase-1.5 backlog item.

### Snapshot rebless workflow
A `zig build update-snapshots` UX that diff-displays before locking. Phase 1
already has `-Dupdate-snapshots=true`; UX improvement is post-1.5.

### Recovery resumption marker
A marker injected into the recovered AST indicating "parser synced here"
would let recovery tests assert *exactly* which sync boundary fired. Worth
considering for Phase 2 but adds AST payload churn; deferred.

### Refactoring `recovery_corpus.zig` into a table-driven test
Currently 117 lines with inline logic; a table-driven version would be
cleaner once the pins are added. Plan as a code-cleanup task INSIDE Phase
1.5 only if the pin extension itself reveals the duplication is painful;
otherwise leave for a future refactor.

### Diagnostics JSON byte-equality in determinism test
Recommended but not locked; planner may choose to include alongside the AST
JSON check in REQ-phase-1.5-3 since marginal cost is near-zero.

## Phase 1.5 Post-Completion Invariants

Added 2026-05-21 by Plan `01.5-05-tests-showcase-submodule-and-verify-harden-PLAN.md`
after the False-GREEN incident diagnosed in `01.5-UAT.md` and

binding ADR.

### Binding invariant

**Phase N exit verification MUST be performed from a freshly-created git
worktree (or ephemeral CI runner) that materializes all inputs only via
committed git state plus `git submodule update --init --recursive`. The main
developer checkout is NEVER an acceptable verification site for a phase-exit
gate. The mechanical implementation for Phase 1.5 is `zig build
phase-1.5-gate-fresh`; future phase gates (phase-2-gate, phase-3-gate, ...)
MUST add an analogous `-fresh` sibling step.**

### Authoritative gate command

```bash
zig build phase-1.5-gate-fresh
```

This step (defined in `build.zig`) shells out to
`scripts/verify-fresh-worktree.sh`, which creates an ephemeral worktree under
`$TMPDIR`, runs `git submodule update --init --recursive` inside it, and runs
`zig build phase-1.5-gate` from the ephemeral tree. Only COMMITTED state can
influence the result.

### Scope of binding

The invariant binds future phases 2 through 6 (and any successor milestone).
Whoever plans Phase 2 — and every subsequent phase planner — MUST include a
`phase-N-gate-fresh` build step in the Phase-N plan, mirroring this pattern.
A phase-exit verification record produced without a `phase-N-gate-fresh` log
line should be rejected by the auditor.
