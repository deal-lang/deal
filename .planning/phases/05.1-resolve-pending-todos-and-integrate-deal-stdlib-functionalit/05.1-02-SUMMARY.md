## What Was Built

**`cli/tests/e2500_cli.rs`** — three new `#[test]` functions:

- `test_e2501_mixed_unit_same_dimension`: fixture `src/mixed_mass_units.deal` uses `kg(1) + g(500)` to trigger E2501 (two distinct Mass-dimension units combined without explicit conversion). Requires `g` in MINIMAL_STDLIB.
- `test_e2502_unknown_unit_literal`: fixture `src/unknown_unit.deal` uses `lb(500)` — a unit not declared in MINIMAL_STDLIB — to trigger E2502 (unknown unit literal).
- `test_e2503_conversion_type_mismatch`: fixture `src/mass_for_voltage.deal` uses `to_g(V(800))` to trigger E2503 (conversion call with wrong source dimension). Fixture filename is `mass_for_voltage.deal`, distinct from `07-conversion-mismatch.deal`.

**MINIMAL_STDLIB extensions:**
- `g <<specializes>> Mass { si_factor = 0.001 }` — gram unit for E2501 mixed-unit scenario
- `to_g <<specializes>> Mass { si_factor = 0.001 }` — conversion target (starts with `to_` prefix; sema detects as conversion call per D-57) needed for E2503

## Deviations from Plan

### Discovery During Execution

**1. [Rule 1 - Bug] E2503 fixture needed `to_<unit>` conversion call, not direct assignment**

- **Found during:** Task 1 GREEN phase
- **Issue:** The plan's suggested E2503 fixture `attribute voltage : Voltage = kg(800)` triggers E2500 (dimension mismatch on direct assignment), not E2503. E2503 is specifically emitted for conversion calls whose callee name starts with `to_` (per `src/sema.zig:490`). Direct assignment of a wrong-dimension unit always produces E2500 regardless of the declared type.
- **Fix:** Added `to_g <<specializes>> Mass` to MINIMAL_STDLIB (a unit whose name starts with `to_`) and used `to_g(V(800))` as the E2503 fixture. The `to_g` unit is structurally identical to `g` but carries the `to_` prefix that sema uses to identify conversion calls (D-57).
- **Files modified:** cli/tests/e2500_cli.rs (MINIMAL_STDLIB + fixture source)
- **Commit:** 9c6155b
