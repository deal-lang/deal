## Warnings

### WR-01: Unclosed array literal emits E0100 instead of E0402

**File:** `src/expr.zig:285-294`
**Issue:** The `l_bracket` arm in `parsePrimary` loops until `.r_bracket` or `.eof`. When the source ends before `]` is found, the loop exits on `.eof` and then `parser.expect(.r_bracket)` at line 290 fires `E0100` ("unexpected token") pointing at the EOF span. The dedicated code `E0402` (`e_unclosed_bracket`) is never emitted by this path. Callers parsing `verification { accepts: [test, simulation }` (missing `]`) will receive an opaque error at the `}` rather than a diagnostic pointing back to the opening `[`.

Existing behaviour elsewhere in the codebase (`e_unclosed_brace` E0401, `e_unclosed_bracket` E0402) establishes the pattern of emitting the dedicated code with the span of the unclosed delimiter. The new array path omits this.

**Fix:**
```zig
// In parsePrimary, l_bracket arm, after the while loop exits:
if (parser.peek().tag == .eof) {
    // Emit E0402 with the span of the opening '[' for a useful diagnostic.
    parser.diags.append(parser.arena, .{
        .code = diagnostics_mod.Codes.e_unclosed_bracket,
        .severity = .err,
        .message = "unclosed `[` — array literal is missing `]`",
        .span = tok.span, // tok is the l_bracket token captured before the loop
    }) catch {};
    // fall through to expect(.r_bracket) which will record EOF span too
}
const close = try parser.expect(.r_bracket);
```

### WR-02: Stale doc comment in `test_e2503_conversion_type_mismatch` describes wrong scenario

**File:** `cli/tests/e2500_cli.rs:367-369`
**Issue:** The doc comment for `test_e2503_conversion_type_mismatch` contains:
```
/// Scenario (fresh minimal fixture — NOT 07-conversion-mismatch.deal):
///   import deal.std.units.{kg, V};
///   attribute voltage : Voltage = kg(800);  ← Mass unit used for Voltage → E2503
```
This describes an E2500 scenario (direct wrong-dimension assignment), NOT the E2503 scenario that the test actually implements. The actual fixture at lines 385-395 uses `attribute mass : Mass = to_g(V(800))` — a conversion call with a wrong-dimension argument — which is what correctly triggers E2503. The stale comment would mislead a developer debugging this test or extending the suite: the described fixture would emit E2500 when they try it, not E2503.

**Fix:**
```rust
/// Scenario (fresh minimal fixture — NOT 07-conversion-mismatch.deal):
///   import deal.std.units.{to_g, V};
///   attribute mass : Mass = to_g(V(800));
///       ↑ to_g converts to Mass, but argument V(800) has Voltage dimension → E2503
```

### WR-03: Array literal silently accepts trailing commas; inconsistent with argument-list strictness

**File:** `src/expr.zig:288`
**Issue:** The `l_bracket` array-literal loop at line 288 silently consumes trailing commas:
```zig
if (parser.peek().tag == .comma) _ = parser.advance();
```
`[test,]` parses without any diagnostic. By contrast, `parseArgList` (expr.zig:411-436) explicitly rejects trailing commas with `E0122`. This inconsistency means:
- `f(test,)` → E0122 (trailing comma rejected)
- `[test,]` → silently accepted

Per the DEAL formatting invariant #5 documented in fmt.zig: "NO trailing commas (E0122 guard)". The guard is only applied at the call-arg level, not the array-literal level, creating a grammar surface where roundtripping through `deal fmt` produces `[test]` from `[test,]` (formatter emits no trailing comma), but the formatter's output no longer round-trips to the same source — a round-trip invariant violation.

**Fix:**
```zig
// After items.append, replace unconditional comma advance with the
// strict pattern used in parseArgList:
if (parser.peek().tag == .comma) {
    _ = parser.advance(); // consume ","
    // After consuming comma, if next is ']', this was a trailing comma.
    if (parser.peek().tag == .r_bracket) {
        parser.diags.append(parser.arena, .{
            .code = diagnostics_mod.Codes.e_empty_arg_comma,
            .severity = .err,
            .message = "trailing comma in array literal is not permitted",
            .span = ..., // span of the consumed comma
        }) catch {};
        break; // close the array cleanly
    }
}
```

### IN-01: `E0110` (`e_expected_expression`) is defined but never emitted

**File:** `src/expr.zig:296-303`, `src/diagnostics.zig:81`
**Issue:** The `else` arm of `parsePrimary` silently creates an identifier node from any unexpected token without emitting any diagnostic:
```zig
else => {
    // Unexpected token — return an identifier node for error recovery.
    const name = parser.source[tok.span.start..tok.span.end];
    return makeNode(allocator, .identifier, tok.span, .{
        .identifier = .{ .name = name },
    });
},
```
The code `E0110` (`e_expected_expression`) is registered in `Codes` at `diagnostics.zig:81` but has zero callsites (`grep -rn "E0110\|e_expected_expression"` returns only its definition). Any unexpected token in expression position is silently accepted as a phantom identifier. This is a pre-existing issue not introduced in phase 05.1, but it is worth tracking because the new `l_bracket` path (which reaches `parsePrimary` for its inner items) relies on the same silent recovery.

**Fix (future):** Emit E0110 in the `else` arm before creating the recovery identifier node:
```zig
else => {
    parser.diags.append(parser.arena, .{
        .code = diagnostics_mod.Codes.e_expected_expression,
        .severity = .err,
        .message = "expected expression",
        .span = tok.span,
    }) catch {};
    const name = parser.source[tok.span.start..tok.span.end];
    return makeNode(allocator, .identifier, tok.span, .{
        .identifier = .{ .name = name },
    });
},
```

### IN-02: E2503 check silently skips non-call arguments to conversion functions

**File:** `src/sema.zig:1207-1230`
**Issue:** `checkCallDimension` only fires E2503 when the single argument to a `to_<unit>` conversion call is itself a `call` node (`if (arg.payload == .call)`). Non-call arguments — identifiers, member-access expressions, binary expressions — are silently skipped:
```zig
if (arg.payload == .call) {
    // ... dimension check ...
}
// falls through silently if arg is not a call node
return;
```
Example: `to_g(someVoltageVar)` where `someVoltageVar` is an identifier with a Voltage-typed value will NOT emit E2503. This is a partial check that could give users a false sense of safety. The check correctly covers the use case proven in the E2503 regression tests (`to_g(V(800))`), but the coverage gap is undocumented. This is a pre-existing limitation not introduced in phase 05.1 but surfaces due to the new E2503 test coverage added in Plan 02.

**Fix (future):** Extend the check to resolve identifier dimensions via the symbol table. Not in scope for phase 05.1 but should be tracked for Phase 5 or 6 semantic completion.
