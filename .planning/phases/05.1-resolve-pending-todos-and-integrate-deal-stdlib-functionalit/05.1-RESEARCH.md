## Architecture Patterns

### Established Pattern: `__array__` synthetic call node (to be REPLACED by D-04)

The current encoding: both `parser_dealx.zig:809` and `parser_deal.zig:1994` produce:
```zig
// call(__array__, [items]) — synthetic, not a real function call
const callee = try p.makeNode(.identifier, open.span, .{
    .identifier = .{ .name = "__array__" },
});
return p.makeNode(.call, span, .{
    .call = .{ .callee = callee, .args = try items.toOwnedSlice(p.arena) },
});
```
`fmt.zig:755-768` undoes this encoding on output. The AST snapshot JSON contains `"k":"call","callee":{"k":"identifier","name":"__array__"}` for every array literal.

**D-04 replacement pattern:** Add a new `NodeKind` variant (e.g., `.array_literal`) to `ast.zig` with payload `struct { items: []*Node }`. Add a `l_bracket` arm to `expr.zig:parsePrimary`'s switch, delegating both existing call sites to it. Add json.zig serialization `"k":"array_literal","items":[...]`. Remove `__array__` special-case from `fmt.zig`.

### Established Pattern: Clone-Before-Free across FFI (`05-LEARNINGS.md`)

Before `deal_free(handle)`, clone diagnostic bytes:
```rust
// cli/src/main.rs:401-420
let diag_json_owned: Vec<u8>;
// ...clone while arena is alive...
diag_json_owned = std::slice::from_raw_parts(out_ptr, out_len).to_vec();
// THEN free
ffi::deal_free(handle);
```
This pattern is already applied; any new E25xx CLI test fixtures must use the same tempdir + minimal stdlib seed pattern from `cli/tests/e2500_cli.rs:87-98`.

### Established Pattern: stdlib seed in CLI tests

To activate the E25xx path in CLI integration tests, create a tempdir with:
```
<tempdir>/
  .deal/
    deps/
      deal-stdlib/
        packages/
          units/
            units.deal   ← minimal stdlib seed (define dimensions + units needed)
  src/
    test_file.deal        ← the test fixture
```
Then invoke `deal check` with `current_dir(tempdir.path())` so D-49 dep collection finds the stdlib seed. See `cli/tests/e2500_cli.rs:87-98` for the exact implementation.

### Anti-Patterns to Avoid

- **Introducing `deal_index_json` calls on `deal_check_with_stdlib` handles:** causes SIGSEGV (Phase 5 landmine). `cli/src/main.rs:427-433` correctly guards against this.
- **Silent token skipping in parser:** the existing `parser_deal.zig:1914-1916` and `1956-1959` patterns are the target of removal; do not replicate them.
- **Two-pass parser changes:** Waves B and C both touch `parser_deal.zig` and `expr.zig`. The planner should consider whether to combine them into one plan to avoid duplicating test runs and diffs over the same files.

## Common Pitfalls

### Pitfall 1: D-04 touches `json.zig` + snapshots — order matters
**What goes wrong:** Implementing the `ArrayLiteral` node in `ast.zig` and updating parsers, but forgetting to add the serialization branch in `json.zig`. The snapshot tests then fail with a panic or incorrect JSON output.
**How to avoid:** Update `json.zig` atomically with `ast.zig`. Run `zig build test -Dupdate-snapshots=true` only AFTER both `ast.zig` and `json.zig` are updated, then immediately re-run WITHOUT the flag to confirm byte-stability.

### Pitfall 2: `deal_index_json` on stdlib-check handles
**What goes wrong:** Adding E25xx CLI tests that call `deal_index_json` on a `deal_check_with_stdlib` handle → SIGSEGV (exit 139, use-after-free of `stdlib_arena`).
**How to avoid:** CLI integration tests for E25xx never call `deal_index_json`. The existing guard at `cli/src/main.rs:427-433` protects the production path; tests go through the same production path and are therefore safe as long as they don't add new handle consumers.

### Pitfall 3: Step 0 gate ordering
**What goes wrong:** Making Wave B parser changes before Step 0, so the baseline state is unknown. If a showcase file was already failing silently, the change appears to cause a regression.

**What goes wrong:** Adding an E2503 CLI integration test that uses the same fixture as the Phase-4 carryover regression pin (`sema.dimensional.regression_pins`, `07-conversion-mismatch.deal`), causing test entanglement.
**How to avoid:** E2503 CLI test uses a fresh minimal fixture in the e2500_cli.rs tempdir pattern, not the showcase regression pin file. The Phase-4 carryover pin tests the Zig sema layer; the D-09 CLI test proves the CLI routing layer.

### Pitfall 5: Spec push without gitlink bump
**What goes wrong:** Wave A spec edits committed to the `spec` repo but the parent `deal` gitlink not bumped. Fresh-worktree gate checks out the old pin and misses the fixes.
**How to avoid:** D-06 explicitly requires: push spec branch → bump gitlink in parent repo → push parent. Verify with `git submodule status` in a fresh worktree.
