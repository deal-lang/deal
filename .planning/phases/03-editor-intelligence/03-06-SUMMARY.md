# Phase 3 Plan 06: Release Pipeline for deal-lsp + vscode-deal Summary

**One-liner:** Closes REQ-phase-3-4-vscode-lsp binary-distribution half by adding a 4-job CI pipeline that cross-compiles `libdeal.a` for 5 targets via Zig (D-53), links `deal-lsp` natively on each matrix runner, packages a main `.vsix` (Marketplace auto-download path per D-50) plus 4 per-platform offline `.vsix` variants with bundled binaries (D-51), and uploads everything with a SHA256SUMS manifest to GitHub Releases. The vscode-deal extension's `bootstrap.ts` consumes those artifacts at runtime — pinned URL `github.com/deal-lang/deal/releases/download/v${DEAL_LSP_VERSION}/deal-lsp-${triple}.tar.gz`, SHA-256 verified against a manifest patched by CI at `vsce package` time (D-52 lockstep), cached under `globalStorageUri`, chmod +x on Unix. The Marketplace `vsce publish` job is OMITTED from the active workflow (lives as commented-out stub) because the required-reviewer Environment gate that would protect `VSCE_PAT` is unavailable on FREE+private repos; the first v0.3.0 publish is done manually from the user's local machine per the runbook below.

    4. Cached-download check: if dest exists AND `sha256File(dest)` matches `SHA256_MANIFEST[triple]`, return dest.fsPath. On mismatch, delete + fall through (handles cache corruption + version upgrades).
    5. First-run dialog via `vscode.window.showInformationMessage(Download / Cancel)`. Cancel throws an Error.
    6. Download with `vscode.window.withProgress` (notification, non-cancellable).
    7. URL = `https://github.com/deal-lang/deal/releases/download/v${DEAL_LSP_VERSION}/deal-lsp-${triple}.tar.gz`.
    8. Extract into a temp staging dir built via `Uri.joinPath(Uri.file(os.tmpdir()), 'deal-lsp-stage-${pid}-${ts}')`; atomic-rename the binary into dest (EXDEV fallback to copy+unlink).
    9. SHA-256 verify; mismatch → delete dest and throw (`SHA-256 mismatch for deal-lsp ${triple}: expected … got …`).
    10. `fs.chmodSync(dest, 0o755)` on Unix.
  - **CWE-22 taint-free design**: every filesystem path built via `vscode.Uri.joinPath` from VS Code-supplied URIs + compile-time string literals; `BINARY_FILENAME` is a frozen Record keyed by the hardcoded triple union (no user input ever reaches a Node `path.join`). ### Task 2 (commits `0381953` vscode-deal + `daf6c4a` deal) — release.yml CI pipeline + SHA256SUMS.template + patch-bootstrap-sha.js

**`vscode-deal/scripts/patch-bootstrap-sha.js`** (137 LOC) — dependency-free Node stdlib build-time script invoked by the release.yml `package-vsix` job:
1. Takes `<artifacts-dir>` as argv[2] (the download-artifact step's output layout: `artifacts/deal-lsp-<triple>/deal-lsp[.exe]` x5).
2. For each triple in `[darwin-arm64, darwin-x64, linux-x64-gnu, linux-x64-musl, win-x64]`, streams the binary through `crypto.createHash('sha256')` and captures the hex digest.
3. Reads `vscode-deal/src/bootstrap.ts` as a string; replaces each `'<triple>': '<sha256-populated-by-CI-on-release-tag>'` entry with `'<triple>': '<real-sha>'`.
4. Validates that exactly `TRIPLES.length` (5) replacements happened.
5. Validates that exactly 1 placeholder occurrence remains (the doc-comment that describes the substitution pattern — any other survival means a SHA256_MANIFEST entry was missed).
6. Writes bootstrap.ts back. Exit 1 on any validation failure to refuse shipping a .vsix that can't SHA-verify the auto-downloaded binary.

End-to-end smoke-tested with synthetic artifacts: 5 hashes substituted from `echo "fake-binary-${triple}"` fixtures, exit 0, original bootstrap.ts restored without diff. Initially failed because the first version's "no placeholders should remain" assertion was too strict (one doc-comment occurrence is legitimate) — fixed via Rule-1 bug fix during execution.

## Manual Marketplace Publish Runbook

After CI Job 4 (`publish-release`) has completed for tag `vX.Y.Z` and uploaded `vscode-deal.vsix` + the 4 offline variants + `SHA256SUMS` to the GitHub Release, the user runs the following from their local machine:

```bash
# 1. Download the main .vsix from the GH Release.
gh release download vX.Y.Z --pattern 'vscode-deal.vsix' --dir ./dist

# 2. Verify SHA-256 against the published SHA256SUMS file.
gh release download vX.Y.Z --pattern 'SHA256SUMS' --dir ./dist
(cd ./dist && shasum -a 256 -c <(grep 'vscode-deal\.vsix' SHA256SUMS))

# 3. Publish to the Marketplace using the local VSCE_PAT (kept in the
#    user's password manager — NOT consumed by CI in this variant).
export VSCE_PAT='<paste from password manager>'
npx vsce publish --packagePath ./dist/vscode-deal.vsix --pat "$VSCE_PAT"

# 4. Repeat for each per-platform offline .vsix variant:
for triple in darwin-arm64 darwin-x64 linux-x64 win32-x64; do
  gh release download vX.Y.Z --pattern "vscode-deal-${triple}-offline.vsix" --dir ./dist
  npx vsce publish --packagePath "./dist/vscode-deal-${triple}-offline.vsix" --pat "$VSCE_PAT"
done
```

**When to re-enable the automated Marketplace job:** the commented-out `publish-marketplace` stub at the bottom of `release.yml` (lines 412-455) is ready to un-comment once any of these is true:
1. The repo is upgraded to GitHub Pro/Team/Enterprise (enables required-reviewer Environments on private repos).
2. The repo is made public (enables required-reviewer Environments on the FREE plan).
3. An equivalent out-of-CI manual-approval mechanism is wired (e.g., human-driven workflow_dispatch on a separate `publish.yml` that runs only after the user has reviewed the .vsix locally).

Re-enable steps are documented in the release.yml header comment immediately above the commented stub.

- **Issue:** The first iteration of `auto_download_url_correct` did NOT inject `extractTarball`. - **Fix:** Added a no-op `extractTarball` injection to the URL test so the failure mode is deterministic (SHA-256 mismatch on the empty buffer) and the URL-capture assertion is tested independently of the extractor.
- **Files modified:** `vscode-deal/src/test/suite/auto_download.test.ts` (single iteration before RED commit)
