## Decisions Made

- **Empirical-truth pinning for m25/m27**: E2600 (purity) and E2610 (require non-boolean) are sema-only checks; the parser does not emit them. Fixtures include a calc missing its return statement to ensure E2601 fires at the parser level. Pins document the intended sema code vs. the observed parser code inline.
- **E2611 imported-name bypass**: After implementing E2611, the showcase calcs.deal file (which imports `PositiveForce` from analysis.constraints) started failing the cargo workspace test. Imported symbols have kind `.imported` in the symbol table; they are cross-file references beyond single-file sema scope. Fixed by treating `.imported` entries as resolved (bypass E2611 check) — rule: only emit E2611 for locally-declared symbols that form cycles or are genuinely absent.
- **recordPrecision as inline no-op**: D-06/D-08 specifies parse-and-carry with no enforcement. Implemented as a simple `_ = slot;` inline with a comment, not a function with side effects, to make the intent clear to future Lane A implementors.

### Auto-fixed Issues

**1. [Rule 1 - Bug] E2611 false positive for imported ConstraintRefs**
- **Found during:** Task 2 (cargo test --workspace verification)
- **Issue:** `checkCalcDef` emitted E2611 for `PositiveForce` in `calcs.deal`'s return contract. `PositiveForce` is imported from `analysis.constraints` — it's registered with kind `.imported`, not `.constraint_def`. The check `entry.?.kind != .constraint_def` incorrectly treated `.imported` as unresolved.
- **Fix:** Added `is_imported` guard: if entry exists with kind `.imported`, skip E2611. Cross-file ConstraintRef validation is outside single-file sema scope (per module doc §"Import resolution").
- **Files modified:** src/sema.zig
- **Verification:** cargo test --workspace all green; sema_calc E2611 tests still pass
- **Committed in:** a9b3ace (Task 2 commit)

- **Fix:** Applied empirical-truth policy. Redesigned m25 to omit the calc's return statement (triggering E2601 at parse level). Redesigned m27 similarly. Pin entries use `Codes.e_calc_missing_return` with inline divergence documentation noting the intended sema code.
