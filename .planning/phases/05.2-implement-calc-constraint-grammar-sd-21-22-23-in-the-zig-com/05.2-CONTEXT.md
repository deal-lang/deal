### Precision vocabulary ‚Äî generalized (amends SD-23)
- **D-06:** A precision contract **`=> ContractItem ( "," ContractItem )*`** (where `ContractItem` = `sig INTEGER` | `¬± Tolerance` | `ConstraintRef`) attaches to **any result-valued declaration**, not just calc returns:
  - **calc return** ‚Äî `calc def F(...) : T => sig 4 { ... }` (original SD-23)
  - **attribute/feature value** ‚Äî `attribute mass : Mass = kg(5) => sig 4;` / `derived attribute q : Pressure = ... => ¬± percent(1);`
  - **constraint `require` value/threshold** ‚Äî precision on the *result value* of a require expression
  - **expression results: result-only** ‚Äî a precision spec binds to the **result** of an expression, **never to bare sub-literals** (rationale: precision is a property of a value, not of its intermediates; literal-level binding is ambiguous and would force propagation semantics).
- **D-07:** **`sig` is non-reserved / contextual** ‚Äî the lexer always emits IDENT for `sig`; the parser recognizes it as a precision keyword ONLY in a precision-contract position (after `=>`). `sig` stays usable as an ordinary identifier (signal/signature) everywhere. (`calc`/`return`/`require` remain **global** reserved words, like existing `part`/`constraint`/`requirement`.)
- **D-08:** **Parse-and-carry only** (locked D4) ‚Äî one **generic AST/IR `precision{kind,value}` slot** shared by ALL attach points; values null/unenforced until numeric-model Lane A. No storage selection, error propagation, or FP-determinism this phase. Keep `spec/ir/v0/schema.json` in sync with the generic slot.
- **D-09:** `¬±` canonical, `+/-` ASCII alias normalized to `¬±` by `deal fmt` (locked D9). `=>` reuses existing `ARROW` token; grammar-architect/planner MUST verify FIRST-set disjointness across all attach points (after `=>`: `{` ‚Üí `.dealx` satisfy block; `sig`/`¬±`/IDENT-ConstraintRef ‚Üí precision contract ‚Äî design ¬ß5.1).

### calc/constraint semantics & diagnostics
- **D-10:** Reserve a **fresh `E26xx` diagnostic range** for calc/constraint/precision (E2500/E2501 are taken by stdlib dimensional checks; E2600+ is unused). Must-have sema checks: **purity rule** (D7 ‚Äî in-only params + single return = expression-valued; any `out` param = statement-valued; error on expression-position call to a statement-valued calc), **dimensional check** of return expr vs declared return type and args vs param types (reuse `sema_dimensional`), **`require`/ConstraintRef resolution** type-checking to Boolean.

### out/inout parameters & multi-output ‚Üí deal-sim
- **D-11:** Parser **accepts `out`/`inout`** direction prefixes in `ParameterDecl` (KerML-fidelity superset hook, design D6/¬ß6). A calc with any `out` param is statement-valued (purity D7). The **in-language assignment-style out-param body stays a clean parse error (E26xx)** ‚Äî calc bodies remain `LocalBinding* ReturnStatement`. Matches the locked design + the showcase (which ships no out-param example).
- **D-12:** **Multi-output compute routes to deal-sim.** Locked direction: an out-param `calc` is a **typed interface** (signature only); **deal-sim provides the body**. No in-language assignment body is planned. THIS phase must **shape the IR `calc_def` node's directed-params + result-features to coincide *exactly* with how a deal-sim's I/O contract maps to KerML** (Function with directed parameters + result features), so a deal-sim can later **collapse into a calc with zero representational rework**. The collapse/binding *mechanism* is a forward phase (Deferred).

<specifics>
## Specific Ideas

- **"Precision applies to any numerical value."** The user's driving principle: significant-figures / tolerance is a property of a *result value*, so it must attach wherever a value is declared (attribute/feature values, require thresholds, calc returns) ‚Äî not be confined to calc return contracts. But **only on the result**, never on bare literals inside a formula (D-06).
- **"Collapse a deal-sim into a calc."** The user's unification: a multi-in/multi-out deal-sim and an out-param calc are the *same KerML shape* (Function with directed params + result features). Make the IR representation identical so a sim's typed I/O contract IS a calc signature ‚Äî the calc declares the interface, deal-sim provides the body (D-12). This resolves the deferred out-param-body question in favor of "no in-language assignment body ‚Äî route to deal-sim."
- **`sig` must stay usable** as an ordinary identifier (the user explicitly rejected reserving it globally) (D-07).

</specifics>
