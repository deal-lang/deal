# Phase 05.2: Code Review Report

**Reviewed:** 2026-06-10T23:30:00Z
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Phase 05.2 implements the calc/constraint grammar (SD-21/22/23) end-to-end through
the Zig pipeline (lexer → AST/parser → fmt/sema → IR/lowering) and the Rust editor
tier (SysML codegen, LSP semantic tokens/hover/definition). The implementation is
structurally sound: the UTF-8 ± dispatch is bounds-safe, the exhaustive switches in
`json.zig` and `fmt.zig` are complete, the constraint-ref cycle detection is
correctly bounded, and the D-07 contextual `sig` is enforced only after `=>`.

One critical correctness bug was found: the LSP hover renderer (`hover.rs`) reads
`int_literal` value using JSON key `"value"` but the actual AST JSON serialization
emits the field under key `"text"`. The tests pass only because they use hand-crafted
synthetic JSON with the wrong key. This means `sig N` precision values always render
as `"?"` in live hover output.

Three warnings cover: the `parseParamList` error-recovery gap that can produce
cascading parse errors, the unreachable `tolerance_absolute` enum variant, and the
wave-0 gate's parse-fails inversion passing vacuously when the binary is absent.
Three info items note dead code and comments that may mislead future maintainers.

## Critical Issues

### CR-01: hover.rs `render_contract_item` reads wrong JSON key for `int_literal` value

**File:** `lsp/src/hover.rs:296-300`

**Issue:** The hover renderer retrieves a `precision_spec` item's value with:
```rust
item.get("value")
    .and_then(|v| v.get("value"))   // <-- looks for key "value" on int_literal node
    .and_then(|v| v.as_str())
    .unwrap_or("?")
```
The outer `.get("value")` correctly retrieves the value *node* (an `int_literal` or
`call` node). The inner `.get("value")` then looks for a key named `"value"` on that
node. However, `json.zig:599-602` serializes `int_literal` with key `"text"`, not
`"value"`:
```zig
.int_literal => |p| {
    try buf.appendSlice(allocator, ",\"text\":");
    try writeStr(allocator, buf, p.text);
},
```
So for `sig 4` the actual AST JSON node is `{"k":"int_literal","span":[...],"text":"4"}`.
The inner `.get("value")` finds nothing, returns `None`, and the hover renders `"sig ?"` 
instead of `"sig 4"`.

The unit tests in `hover.rs` (lines 750-774) pass only because they use synthetic JSON
that sets `"value": "4"` directly on a fake `"integer"` node — a shape that never
appears in real parser output.

The same bug affects the `tolerance_relative`/`tolerance_absolute` case for `±`
expressions: when the value is a `call` node (e.g. `percent(1)`), the inner
`.get("value")` also fails since `call` nodes have no `"value"` field. For complex
expressions the fallback `"?"` is arguably acceptable (the value is `"<expr>"` in IR
anyway), but for the `sig N` case with a literal integer, the wrong key is definitely
a bug.

**Fix:** Change the key from `"value"` to `"text"` for the inner access on `int_literal`
nodes. Either add a branch for the `int_literal` shape or, more robustly, try both keys:
```rust
"precision_spec" => {
    let prec_kind = item.get("kind").and_then(|v| v.as_str()).unwrap_or("");
    let val_node = item.get("value");
    let value = val_node
        .and_then(|v| v.get("text").or_else(|| v.get("value")))
        .and_then(|v| v.as_str())
        .unwrap_or("?");
    match prec_kind {
        "sig_figures" => format!("sig {value}"),
        _ => format!("± {value}"),
    }
}
```
Also update the unit test at line 766 to use `"text": "4"` (matching the real AST shape)
to prevent this class of test/reality mismatch recurring.

## Warnings

### WR-01: `parseParamList` has no forward-progress guard on error recovery

**File:** `src/parser_deal.zig:1219-1233`

**Issue:** When `parseParamDecl` encounters an unexpected token (e.g. a double comma
`,,` or a stray keyword inside a parameter list), `expect(.ident)` emits a diagnostic
and returns the wrong token WITHOUT advancing the parser past it. The `parseParamList`
loop `while (p.peek().tag != .r_paren and p.peek().tag != .eof)` then immediately
calls `parseParamDecl` again on the same token. Since neither `parseParamDecl` nor
the loop itself has a forward-progress guard, the parser will spin consuming one token
per iteration until it eventually hits `r_paren` or `eof`, producing one E0100
diagnostic per stuck token.

This differs from `parseCalcBody` and `parseConstraintBody` (lines 1404-1433,
1511-1530), which were explicitly patched (Plan 03 deviation #2) with forward-progress
guards after identical infinite-loop bugs were found.

The bug does not cause a hang (each `expect()` call consumes exactly one token), but
it produces a cascade of misleading E0100 diagnostics on malformed parameter lists, and
the resulting `param_list` node contains corrupt `param_decl` entries (each with the
wrong token text as the name).

**Fix:** Add a forward-progress guard to the `parseParamList` loop, mirroring the
pattern in `parseConstraintBody`:
```zig
while (p.peek().tag != .r_paren and p.peek().tag != .eof) {
    const loop_start = p.peek().span.start;
    const param = try parseParamDecl(p);
    try params.append(p.arena, param);
    if (p.peek().tag == .comma) {
        _ = p.advance();
    } else if (p.peek().tag != .r_paren and p.peek().tag != .eof) {
        // No comma and no close paren — we may be stuck on an error token.
        // Only add a guard if position hasn't advanced since the loop start.
        if (p.peek().span.start == loop_start) _ = p.advance();
    }
}
```

### WR-02: `PrecisionKind.tolerance_absolute` is unreachable dead code

**File:** `src/ast.zig:633`, `src/fmt.zig:865`, `src/lowering.zig:401-404`

**Issue:** The `PrecisionKind` enum declares three variants:
```zig
pub const PrecisionKind = enum { sig_figures, tolerance_absolute, tolerance_relative };
```
However, neither `parsePrecisionSpecPlusMinus` (line 1321-1328) nor
`parsePrecisionSpecPlusSlashMinus` (lines 1330-1338) ever assigns `.tolerance_absolute`.
Both assign `.tolerance_relative`. The variant `tolerance_absolute` is therefore
unreachable through any parser path.

This means: (a) `fmt.zig` groups them together
(`tolerance_absolute, tolerance_relative =>`) — dead but harmless; (b)
`lowering.zig` maps `tolerance_absolute` to `"tolerance_absolute"` — a string that
will never appear in lowered IR; (c) if a future consumer branches on `tolerance_absolute`
expecting it to signal a distinct semantic from `tolerance_relative`, they will never
see it.

The issue is not a crash but indicates either the parser is incomplete (the grammar may
intend `±expr` = absolute and `±percent(...)` = relative) or the enum variant should be
removed.

**Fix (option A — remove the unused variant):** Delete `.tolerance_absolute` from the
`PrecisionKind` enum and update all three exhaustive switch arms that reference it.
This makes the type honest.

**Fix (option B — implement the distinction):** If the design intends
`±percent(x)` = `tolerance_relative` and `±x` (bare numeric) = `tolerance_absolute`,
wire the distinction in `parsePrecisionSpecPlusMinus`:
```zig
fn parsePrecisionSpecPlusMinus(p: *Parser) !*ast.Node {
    const pm_tok = p.advance();
    const val = try expr.parseExpression(p, 0);
    // percent(...) call → relative; plain value → absolute
    const kind: ast.PrecisionKind = if (val.kind == .call) .tolerance_relative else .tolerance_absolute;
    const span = ast.Span{ .start = pm_tok.span.start, .end = val.span.end };
    return p.makeNode(.precision_spec, span, .{
        .precision_spec = .{ .kind = kind, .value = val },
    });
}
```

### WR-03: wave-0 gate parse-fails inversion passes vacuously when binary is absent

**File:** `build.zig:683-688`

**Issue:** The wave-0 gate step verifies that `calcs.deal` cannot yet be parsed by
using exit-code inversion:
```zig
"! target/release/deal parse tests/showcase/packages/analysis/calcs.deal"
```
As documented in `05.2-01-SUMMARY.md` (Decisions Made, third bullet), in a fresh
worktree `target/release/deal` does not exist, so the shell command `target/release/deal`
fails with "No such file or directory" (exit code 127), the `!` inverts it to 0, and
the gate step passes. The gate intent (the oracle CANNOT be parsed by the current
compiler) is satisfied — but by accident: the binary is absent, not by the oracle
being unparseable.

The consequence: if someone runs `zig build phase-05.2-wave0-gate` AFTER Wave 1 (the
lexer has been updated, the binary re-built and present), the gate would fail correctly.
But in a CI environment where neither Wave-0 nor a stale binary is present, the
inversion passes silently whether or not the binary exists for the right reason.

This is a build/gate quality problem, not a runtime correctness issue. The Phase 05.2
final gate (`phase-05.2-gate`) does not have this problem since it explicitly builds
the binary first.

**Fix:** Add an explicit binary-existence check before the inversion step, or reframe
the parse-fails assertion to use a different signal (e.g. assert the binary exists AND
fails to parse):
```zig
const gate_052_w0_oracle_parse_fails = b.addSystemCommand(&[_][]const u8{
    "bash",
    "-c",
    // Require binary exists; then require parse fails on the oracle.
    "test -f target/release/deal && ! target/release/deal parse tests/showcase/packages/analysis/calcs.deal",
});
```

### IN-01: `checkCalcDef` in sema.zig has vestigial dead-code comments that misrepresent the implementation

**File:** `src/sema.zig:1006-1034`

**Issue:** Lines 1006-1034 inside `checkCalcDef` contain a dense block of redundant
and occasionally self-contradicting comments that were left from the implementation
process:
```zig
// The parser emits E2601 when no return statement is found; the return_stmt
// is guaranteed to be non-null (parser synthesizes a recovery node).
// Sema re-checks by scanning the body for a require_statement node
// (which doubles as ReturnStatement per the Wave-2 decision).
// Actually per Plan 03 decision...
// ... For now: use the heuristic that if the expression is an identifier
// named "" (empty), emit E2601. The parser's recovery node may be an
// identifier with name "".
_ = return_expr; // used below
```
The actual implementation checks for the name `"<missing>"` (lines 1095-1107), not an
empty string `""`. The intermediate comments describing the empty-string heuristic are
stale and misleading. The `_ = return_expr; // used below` comment is also misleading
— `return_expr` is used at line 1095 as `ret_condition`, not `return_expr` directly
(the variable is re-bound).

**Fix:** Remove or consolidate the stale implementation comments. The final check
logic is clear; a single explanatory comment replacing the block is sufficient:
```zig
// Check #8b: parser synthesizes a require_statement with condition=identifier("<missing>")
// when no return keyword was found and emits E2601 itself. Sema reinforces by
// re-checking the recovery sentinel.
```

### IN-02: `tolerance_absolute` and `tolerance_relative` both map to `"tolerance_absolute"` string in `lowering.zig`

**File:** `src/lowering.zig:401-404`

**Issue:** The `extractPrecisionFromReturnContract` function maps `PrecisionKind` to
its string representation:
```zig
const kind_str: []const u8 = switch (ps.kind) {
    .sig_figures => "sig_figures",
    .tolerance_absolute => "tolerance_absolute",
    .tolerance_relative => "tolerance_relative",
};
```
This is correct as written, but combined with WR-02 (`.tolerance_absolute` is never
produced by the parser), the string `"tolerance_absolute"` will never appear in IR
output. If the design eventually distinguishes the two tolerance kinds, this is the
right shape. As long as WR-02 is not resolved, this dead branch is harmless but
documents an unimplemented distinction.

**Fix:** Resolve after WR-02: if `.tolerance_absolute` is removed from the enum, remove
this arm. If the distinction is implemented, keep it as-is.

### IN-03: `ReturnStatement` struct in `ast.zig:655-657` is declared but never used

**File:** `src/ast.zig:655-657`

**Issue:** The `ast.zig` file declares:
```zig
pub const ReturnStatement = struct {
    expr: *Node,
};
```
This struct is never referenced anywhere in the codebase. The Wave-2 decision (Plan 03
key-decisions) explicitly chose to reuse `require_statement` NodeKind for return
statements rather than introducing a 10th NodeKind. The `ReturnStatement` struct was
declared but the corresponding payload union arm was never added — `calc_body.return_stmt`
is a `*Node` whose payload is `.require_statement`, not `.return_statement`.

The struct has no payload union slot and cannot be instantiated via a Node. It is dead
exported code.

**Fix:** Remove the `ReturnStatement` struct from `ast.zig` since it is unused and its
presence implies a design path that was deliberately not taken. A comment on the
`CalcBody.return_stmt` field explaining the `require_statement` reuse is already
present (line 650) and is sufficient.
