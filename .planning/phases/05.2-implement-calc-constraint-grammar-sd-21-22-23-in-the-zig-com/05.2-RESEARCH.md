## Summary

Phase 05.2 implements the locked calc/constraint grammar design (SD-21/22/23, decisions D1–D10 from spec commit `7ce59bc`) plus two user-locked expansions (generalized precision attach points per D-06; IR calc_def shape aligned with deal-sim I/O per D-12) through the full hand-written Zig pipeline — lexer → parser/AST → fmt/sema → IR/lowering → SysML/KerML codegen — plus editor surfaces (tree-sitter + vscode-deal + LSP).

The phase is structured as six waves, each roughly one GSD plan, gated by the showcase parse oracle (calcs.deal + constraints.deal must exit 0 after Wave 2). The most complex work is Wave 2 (AST + parser), which introduces 7–8 new NodeKinds and requires new parse functions for calc bodies, constraint bodies, parameter lists, and the `=>` return contract. Wave 0 (spec submodule merge) is a blocking precondition before any Zig work begins.

The closest implementation analog is Phase 05.1's `array_literal` end-to-end pipeline thread: a new NodeKind added to `ast.zig → NodeKind enum → Payload union → json.zig:nodekindName → fmt.zig:writeNode → expr.zig/parser_deal.zig`. The primary gotcha carried from that work: Zig's tagged-union Payload requires `NodeKind` enum order to match `Payload union` arm order exactly, and the exhaustive switches in `json.zig` and `fmt.zig` are compile-time enforced — a missing arm is a build error.

**Primary recommendation:** Follow the wave spine exactly: Wave 0 (spec gate with 3-way hand-merge) → Wave 1 (lexer: 3 reserved words + `plus_minus` token) → Wave 2 (AST + parser: new NodeKinds + parse functions) → Wave 3 (fmt/sema: normalization + E26xx checks) → Wave 4 (IR/lowering/codegen) → Wave 5 (editor: parallelizable after Wave 2). Mirror the `array_literal` pattern for each new NodeKind. Preserve `sema.analyze` calling convention; add new checks to the existing Analyzer pass.

### Pattern 1: Adding a New NodeKind (array_literal template)

**What:** Every new NodeKind requires changes in exactly 5 files (ast.zig, json.zig, fmt.zig, parser_deal.zig, and optionally ir.zig/lowering.zig).
**When to use:** For each of the 8+ new NodeKinds in Wave 2.

The 05.1 `array_literal` path (commit history) proves the exact sequence:

1. `ast.zig` — Add to `NodeKind` enum **at the correct position** (enum order drives Payload union arm order in Zig). Add struct to `Payload union`.
2. `json.zig` — Add arm to `nodekindName()` exhaustive switch (~line 685).
3. `fmt.zig` — Add arm to `writeNode()` exhaustive switch (~line 81).
4. `parser_deal.zig` — Add parse function + dispatch in `parseMember` / `parseDefinition`.
5. `ir.zig` / `lowering.zig` — Add IR counterpart + lowering arm (Wave 4).

**Critical gotcha:** In Zig 0.16.0, the `Payload = union(NodeKind)` requires arm ORDER to match the enum ORDER exactly. If you add `.calc_def` to `NodeKind` enum but insert it in the wrong position in `Payload`, the build fails with a confusing type error. Add enum and union arm in the same relative position.

```zig
// ast.zig: add to NodeKind enum (after constraint_def, before need_def)
calc_def,          // NEW
param_list,        // NEW
param_decl,        // NEW
return_contract,   // NEW
precision_spec,    // NEW
constraint_ref,    // NEW
constraint_body,   // NEW
calc_body,         // NEW
require_statement, // NEW

// ast.zig: add to Payload union in matching order
calc_def: CalcDefinition,
param_list: ParameterList,
// ...

// json.zig: add to nodekindName() (must be exhaustive)
.calc_def => "calc_def",
.param_list => "param_list",
// ...

// fmt.zig: add to writeNode() (must be exhaustive)
.calc_def => |*p| try ctx.writeCalcDef(p, indent),
// ...
```

### Pattern 2: Contextual Keyword Parsing (`sig` pattern, D-07)

**What:** The lexer emits `.ident` for `sig`. The parser recognizes `sig` as a precision-spec keyword only AFTER consuming the `=>` (`.arrow`) token.

```zig
// In parseReturnContract, after consuming .arrow:
fn parseReturnContract(p: *Parser) !*ast.Node {
    const arrow_tok = p.advance(); // consume .arrow
    // ...
    while (true) {
        const tok = p.peek();
        if (tok.tag == .ident and std.mem.eql(u8, p.tokenText(tok.span), "sig")) {
            // treat as precision spec: sig INTEGER_LITERAL
            const contract_item = try parsePrecisionSpec(p);
            // ...
        } else if (tok.tag == .plus_minus) {
            // ± Tolerance
            const contract_item = try parsePrecisionSpecPlusMinus(p);
            // ...
        } else if (tok.tag == .ident) {
            // ConstraintRef: QualifiedName _ArgumentList?
            const contract_item = try parseConstraintRef(p);
            // ...
        }
        // ...
    }
}
```

### Pattern 3: ± Token Lexing (UTF-8 multi-byte)

**What:** `±` is U+00B1, encoded as 0xC2 0xB1 in UTF-8 (2 bytes). The lexer currently falls through to the `else` branch on `c0 >= 0x80` and emits `.unknown`. A new dispatch path is needed.

```zig
// In lexer.zig scanPunctuation, inside the else branch for c0 >= 0x80:
if (c0 == 0xC2 and c1 == 0xB1) {
    self.pos = start + 2;
    return tokenAt(.plus_minus, start, self.pos);
}
```

Also handle the ASCII alias `+/-` as a 3-char sequence that fmt normalizes away:

```zig
// The parser (not lexer) can also accept a token sequence .plus .slash .minus
// and synthesize a precision_spec. OR the lexer can eagerly recognize "+/-" as
// a single plus_minus token by checking c0=='+', c1=='/', c2=='-'.
// Recommendation: lexer recognizes both, fmt ensures ± is always emitted.
```

### Anti-Patterns to Avoid

- **Adding `sig` to `global_keywords`:** D-07 explicitly forbids this. `sig` must remain `.ident` everywhere except after `=>`. If added to global_keywords, existing code using `sig` as an identifier (e.g., a signal variable named `sig`) would break.
- **Redefining `_ArgumentList`:** The grammar says "reuse existing call-argument production." Do not create a new `ParameterArgumentList` production for constraint/calc invocations. The existing `.call` node + `FunctionCallExpression` handles all invocations.
- **Changing existing ConstraintDef = ElementDef alias without audit:** `lowering.zig:elemDefOf()`, `astKindToIrKind()`, and `sysml_v2.rs` all match on `constraint_def` expecting an ElementDef payload. These dispatch sites must be updated when the payload type changes.
- **Skipping json.zig/fmt.zig exhaustive switch arms:** Zig 0.16.0 enforces exhaustive switches over `union(enum)`. A missing arm is a compile error, not a runtime panic. Add new NodeKind arms to both files before first build.
- **Placing `=>` disambiguation in the wrong tier:** After a `calc def ... : T`, `=>` opens a `ReturnContract`. After `.dealx` `[<satisfy>]`, `=>` opens a `SatisfyReturnType`. These are different start symbols and file modes — there is no in-grammar clash, but the Zig parser's mode dispatch must confirm the parser is in `.deal` mode before treating `=>` as `ReturnContract`.

**`src/lexer.zig` — scanPunctuation update for ±:**

```zig
// In scanPunctuation, before the generic high-bit-byte handling:
// ± U+00B1 is encoded as 0xC2 0xB1 in UTF-8.
if (c0 == 0xC2) {
```

**FIRST-set disjointness confirmation:**

After `=>` in `.deal` context:
- `{` → would be a CalcBody start — this cannot happen because `=>` is consumed BEFORE the body; valid FIRST tokens for `ContractItem` are: `sig` (as IDENT), `.plus_minus`, or an IDENT (ConstraintRef)

- `}` → close

- `derived` / `attribute` → LocalBinding
- `@` → AnnotationStatement
- `}` → if no ReturnStatement seen, error E260x

A deal-sim's `inputs` dict IS the set of `in` IrParam entries; the `outputs` dict IS the set of `out` IrParam entries. This structural identity is the "zero rework" guarantee.

## Common Pitfalls

### Pitfall 1: Zig Tagged-Union Field Order Mismatch
**What goes wrong:** Compile error "field 'calc_def' does not match enum variant" or mismatched union arms.
**Why it happens:** `Payload = union(NodeKind)` requires arms in the same order as the enum. Inserting a new NodeKind entry in the enum at position N but inserting the Payload arm at position M > N causes silent structural mismatch or a compile error.
**How to avoid:** Add enum entry and Payload arm in the same relative position in the same commit. Run `zig build` immediately after each addition.
**Warning signs:** Build error referencing union/enum ordering in ast.zig.

### Pitfall 2: Exhaustive Switch Arms Missing in json.zig and fmt.zig
**What goes wrong:** Build fails immediately when a new NodeKind is added to the enum without adding a corresponding arm to `nodekindName()` (json.zig) or `writeNode()` (fmt.zig).
**Why it happens:** Both functions use `switch(node.kind)` over `NodeKind` — Zig 0.16.0 enforces exhaustiveness for `union(enum)`.
**How to avoid:** Add stubs (returning `"unknown"` or `return error.Unimplemented`) as placeholders in the same commit as the NodeKind addition, then fill them in with real implementations.
**Warning signs:** First `zig build` after adding a NodeKind fails with "switch must handle all cases."

### Pitfall 3: ConstraintDef = ElementDef Alias Break
**What goes wrong:** All dispatch sites that match `.constraint_def => |e| e` expecting an `ast.ElementDef` break at compile time when `ConstraintDefinition` becomes a new struct type.
**Why it happens:** Currently `ConstraintDef = ElementDef` is a type alias. After Wave 2, it must become a distinct struct.
**How to avoid:** Search for all uses of `.constraint_def => |e|` in ast.zig, parser_deal.zig, lowering.zig, sema.zig, json.zig, fmt.zig, lsp/src/*.rs before changing the alias. Update all of them in the same Wave 2 plan.
**Warning signs:** Compile errors in `lowering.zig:elemDefOf()` or `sema.zig` when extracting ElementDef fields from a constraint_def payload.

### Pitfall 4: `sig` Accidentally Reserved Globally
**What goes wrong:** Any DEAL source file that uses `sig` as a variable name (e.g., `attribute sig : SignalStrength = ...`) fails to parse with E0101 "expected definition keyword" after `sig` is treated as a keyword.
**Why it happens:** Adding `sig` to `global_keywords` (the easy mistake) promotes it from `.ident` to `.kw_sig` everywhere.
**How to avoid:** D-07 is explicit. Do NOT add `sig` to `global_keywords`. The parser's contextual recognition after `=>` handles it via text comparison of the `.ident` token.
**Warning signs:** Showcase files that use `sig` as identifiers start failing to parse in Wave 1 snapshot tests.

### Pitfall 5: `±` Token Not Recognized in Fresh Worktree
**What goes wrong:** calcs.deal uses `±` (U+00B1, 2-byte UTF-8), which the current lexer emits as `.unknown`. Until Wave 1 adds the `0xC2 0xB1` dispatch, the oracle file fails to lex.
**Why it happens:** The `scanPunctuation` else branch emits `.unknown` for all high-bit bytes except string/template content.
**How to avoid:** Implement the multi-byte dispatch in the same Wave 1 plan as the keyword additions. Run the showcase oracle parse test at the END of Wave 1 to confirm lex is clean.
**Warning signs:** `lexer_snapshot.zig` for calcs.deal shows `.unknown` tokens at the `±` positions.

### Pitfall 6: Arrow Disambiguation Confusion
**What goes wrong:** The `=>` in a calc definition's return contract is mistaken for the `=>` in `.dealx`'s `SatisfyReturnType`.
**Why it happens:** Both use `.arrow` token.
**How to avoid:** These are different start symbols in different file-mode parsers. The `.deal` parser's `parseCalcDef` encounters `=>` after `": Type"` — at this exact parser state, a `{` cannot follow as a ContractItem (the grammar says ContractItem FIRST is `sig`/`±`/IDENT, not `{`). Verify: after consuming `=>`, peek at next token — if it's `{`, that would be an error in the calc contract context.
**Warning signs:** Test the worked example in the spec §8 that shows `calc def Drag(...) : Force => ± percent(1), PositiveForce { ... }` — the `{` opens the CalcBody, not the contract.

### Pitfall 7: IR schema.json Forgot calc_def
**What goes wrong:** `deal_ir_json()` emits a calc_def IR node but schema validation fails because `"calc_def"` is not in the `$defs.NodeKind.enum` array in `spec/ir/v0/schema.json`.
**Why it happens:** The IR Zig code compiles fine; the JSON schema is a separate file that must be kept in sync.
**How to avoid:** Wave 4 must update schema.json in the same plan as `ir.zig`. The `determinism_lower_twice.zig` test exercises IR lowering; add a schema-validation step for the lowered calcs.deal IR.
**Warning signs:** Schema validation failure in golden fixture tests (cargo test --workspace) after Wave 4.
