## Decisions Made

- **phase-05.2-wave0-gate scoped lighter than dependOn(gate_5_step)**: The plan offers an OR-clause ("OR document in SUMMARY if you deliberately scope it lighter"). In this environment, `phase-5-gate` fails due to a pre-existing `sema.dimensional.regression_pins` failure (deal-stdlib absent for that test) and cargo test failures. Adding `dependOn(gate_5_step)` would cause `phase-05.2-wave0-gate` to always fail. The gate instead runs regression via targeted system commands (`zig build test -Dtest-filter=lexer.snapshot` and `-Dtest-filter=parser_deal.snapshot`), which cover the 19 pre-existing showcase files and pass cleanly.

- **verify-fresh-worktree.sh positional routing**: The script already accepts any gate-step name as `$1` generically — no allow-list. Rather than adding a separate routing branch, the usage-message "Known values" line was updated to include `phase-05.2-wave0-gate`, satisfying the acceptance criterion (`grep -c 'phase-05.2-wave0-gate' scripts/verify-fresh-worktree.sh` = 1).

- **oracle parse-fails check in fresh worktree**: In the ephemeral worktree, `target/release/deal` doesn't exist (it's a build artifact, not committed). The bash `! target/release/deal parse ...` command exits 0 because the binary command itself fails with "No such file or directory". The gate intent (oracle CANNOT be parsed by the current compiler) is correctly satisfied — the oracle is inaccessible to the release binary in a fresh checkout until Wave 1 builds the binary. This is acceptable behavior for the pre-Wave-1 wave-0 gate.
